# v1.10.1 — 目录树签名 API（自动刷新性能优化）

> For agentic workers: REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**版本号**：v1.10.1（patch — 性能优化，无破坏性）
**起草时间**：2026-05-11 00:50
**优先级**：P0（v1.10.0 引入的性能负担，需要紧跟修复）

## Goal

把 v1.10.0 自动刷新的网络/CPU 开销降低 **>95%**：从"每 10 秒拉 1.3 MB 全树 JSON"改为"先拉 ~80 字节签名，签名变化时才拉全树"。

## Why

v1.10.0 上线后实测：
- 当前用户的扫描根有 **4001 个节点**（1110 目录 + 1029 HTML + 1862 MD），深度 10
- `/api/tree?refresh=1` 单次返回 **1.3 MB JSON**，耗时 150-200ms
- v1.10.0 默认 10 秒一次 → **每分钟 7.8 MB**，6 分钟 50 MB
- 即使页面后台时已停轮询，前台使用时仍持续吃带宽 + 解析 4001 节点

98% 的轮询其实"什么都没变"，只是为了发现"是否有变化"——这是典型的"贵的全量替代廉价的增量检查"反模式。

## Architecture

经典的 **ETag/HEAD 模式**：

```
Client                   Server
  │                        │
  ├── GET /api/tree-sig ──▶│  扫描所有 HTML/MD 文件
  │                        │  hash(path|mtime|size) for each
  │◀───── {sig:"abc"} ─────┤  返回 80 字节 JSON
  │                        │
  ├── 旧 sig === abc?      │
  │   YES → 跳过           │
  │   NO  → 拉全树 ────────▶│  GET /api/tree?refresh=1
  │                        │
```

### 关键设计

| 决策 | 选择 | 理由 |
|---|---|---|
| 签名算法 | MD5(sorted("path\|mtime\|size" 拼接)) | 快、确定、抗碰撞足够 |
| 包含字段 | path / mtime / size | 改名/改时间/改大小都触发，删除/新增也触发 |
| 不含 | dir 信息 | 文件变化必然触发签名变化，dir 自动跟随 |
| 缓存 TTL | 1 秒（极短） | 给极高频调用兜底，正常 10s 不会撞上 |
| 排序参数 | 不影响签名 | 排序是前端展示问题，签名只反映"内容是否变" |

### 性能预估

| 指标 | v1.10.0 | v1.10.1 | 改善 |
|---|---|---|---|
| 单次请求大小（无变化） | 1.3 MB | ~80 B | **99.99%↓** |
| 单次请求耗时（无变化） | 150-200 ms | ~30-50 ms | 约 4 倍↓ |
| 每分钟流量（10s 周期） | 7.8 MB | ~480 B | **99.99%↓** |
| 浏览器 JS 解析压力 | 解析 4001 节点 | 解析 1 个 sig 字符串 | ≈ 4000×↓ |

只有**真有变化**时才付全树代价，这种"按需"的代价是合理的。

## Risk

| 风险 | 缓解 |
|---|---|
| MD5 碰撞 | 几乎不可能（需要构造特殊 path/mtime/size 组合）|
| sig 计算成本反而高 | 只有 stat() 调用，比构建 dict 树轻 5×+ |
| 缓存 TTL 1s 太短导致连发抖动 | 1s 足够，且每次 stat() 几乎免费 |
| 前端首次轮询不发 tree 怎么办 | 不需要——init() 已经做过 loadTree() 全量加载了，sig 只是"之后判断有没有变" |

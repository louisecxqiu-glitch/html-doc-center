# v1.13.x Bug 修复 · diffTreeAndPatch 双重渲染

## 问题（Problem）

打开 DocCenter 搜索框输入 `step` 时，侧边栏「演化推理链」目录下 6 个 Step 文件**各显示 2 次**，匹配数 17（实际期望 11）。

刷新页面（Cmd+Shift+R）后恢复正常 → 是增量刷新 bug，不是后端 / 全量渲染问题。

**复现条件**：在 DocCenter 已打开的状态下，外部新建一个目录并往里塞 N 个 HTML 文件（mtime 接近）→ 30s auto-refresh 触发 → 该目录及内部全部文件被双重渲染。

## 根因（Root Cause）

`web/app.js` 第 681-742 行 `diffTreeAndPatch` 的双重渲染缺陷。

新建一个目录 `演化推理链/` 内含 6 个新 HTML 文件时：

1. `flatTree` 把新树扁平化 → 7 个新增项进入 `added` 列表（1 个目录 + 6 个文件）
2. `for (const p of added)` 循环按 Map 迭代序处理，父目录通常在子文件之前
3. **处理父目录时**：第 718 行 `renderNode(meta.node, meta.depth)` 会**递归生成包含 6 个 Step 子节点的完整子树**（见 `renderNode` 第 858-859 行的 `for (const child of node.children) children.appendChild(renderNode(child, depth+1))`）
4. **接着处理 6 个 Step 文件**：每个又走一次 718 行 `renderNode` → 找到刚刚被步骤 3 创建的 parentEl → append 到 `parentChildrenContainer`
5. ⇒ 6 个 Step 文件被渲染了 2 次

第 712-713 行原作者注释 `// 父节点也是新增的，等下一轮（实际它会一起出现，所以这里跳过没问题）` 暴露原作者考虑过"子在父之前"的场景但**忽略了"父在子之前"的场景**——后者是 `flatTree` walk 的常见顺序。

## 解法（Solution）

修复策略：**`added` 列表内做祖先剪枝**——若某个新增项的某个祖先路径也在 `added` 集合里，则跳过它（祖先的 renderNode 已经把它一起渲染了）。

最小改动：在第 701 行 `for (const p of added)` 循环的开头加 4 行判断。不改 `renderNode`、不改 `flatTree`、不改增量逻辑主体——保持向后兼容，保留 v1.10.4 的排序修复成果。

```js
const addedSet = new Set(added);
for (const p of added) {
  // 祖先剪枝：若某个祖先目录也在新增列表里，渲染祖先时已经递归生成了 p
  const meta = newFlat.get(p);
  if (!meta) continue;
  let anc = meta.parentPath;
  let skipBecauseAncestorAdded = false;
  while (anc !== null && anc !== undefined) {
    if (addedSet.has(anc)) { skipBecauseAncestorAdded = true; break; }
    const ancMeta = newFlat.get(anc);
    anc = ancMeta ? ancMeta.parentPath : null;
  }
  if (skipBecauseAncestorAdded) continue;
  // ……原有逻辑
}
```

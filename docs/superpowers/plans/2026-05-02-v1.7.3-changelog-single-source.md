# v1.7.3 — CHANGELOG 架构重构（单一数据源）

## Goal

**还清一个长期技术债**：`CHANGELOG.md` 和 `CHANGELOG.html` 两份冗余文件，每次迭代必须手动双写。用户今天问出"为什么不做成 MD 动态渲染"，直接戳到架构不合理处。

重构后：
- `CHANGELOG.md` 成为**唯一数据源**
- `/changelog` 路由返回**壳子 HTML**，前端用已有 `marked.min.js` 渲染 MD
- `CHANGELOG.html` **删除**（git 历史保留）
- SOP 从"**三写铁律**"（MD + HTML + git commit）**降级为"双写铁律"**（MD + git commit）

## Architecture

### 方案 B2（前端解析）

```
/changelog
  └─ 返回 /static/changelog-shell.html（新建壳子，含样式和 JS）
     └─ JS fetch('/api/changelog-raw')
        └─ 服务端返回 CHANGELOG.md 原文（text/plain）
     └─ 前端 marked.parse() → innerHTML
     └─ 后处理：识别 "## [vX.Y.Z] — 日期 · 标题" 转成时间轴卡片
```

### 关键设计

1. **前端渲染**：复用 `web/vendor/marked.min.js`，后端零新依赖
2. **后端变化极小**：`/changelog` 指向新壳子；新增 `/api/changelog-raw` 返回 MD 原文
3. **样式保留**：把现有 `CHANGELOG.html` 的 `<style>` 提取到独立 `changelog.css`，保持时间轴视觉一致
4. **时间轴识别**：CHANGELOG.md 每个版本都以 `## [vX.Y.Z] — 日期` 开头，marked 渲染后用 CSS 选择器或一次性 JS 遍历把相邻的 `<h2>+后续内容` 包装成 `.release-card`

## Tech Stack

原有 marked.min.js（v11 UMD）+ 原生 JS + CSS。后端零新依赖。

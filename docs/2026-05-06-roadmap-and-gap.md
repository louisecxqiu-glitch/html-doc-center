# DocCenter v1.8.2 · 现状盘点 + 距目标多远 + 下一步优化

> 日期：2026-05-06
> 作者：路易乔布斯
> 当前版本：**v1.8.2**（14 个 tag，一路从 v1.0 → v1.8.2）

---

## 1. 产品核心目标（先说清楚）

DocCenter 的**北极星**：

> **让 AI 生成的 HTML 报告，不再是"只读的快递包裹"，而是"能接着改、能积累的工作成果"。**

拆成三条可验证的：

| 目标 | 具体含义 | 检验 |
|---|---|---|
| **G1 · 零仪式编辑** | 任何 HTML 点开就能改，不需要"进入编辑模式" | 用户零学习成本完成一次 1-click 编辑 |
| **G2 · 零数据丢失** | 改了就不会丢，误操作也能回到任何历史状态 | 断电/关窗/误覆盖均可恢复 |
| **G3 · 零 AI 依赖** | 小范围调整不用回去找 AI 重生成 | 90% 的"小改"DocCenter 内完成 |

---

## 2. 当前状态 · 已做到什么程度（v1.8.2 盘点）

### 2.1 G1「零仪式编辑」完成度：**85%**

✅ **已做到**：
- 目录树一屏看所有 HTML（~170 份，首屏 <1s）
- 点开即编辑，不用点"编辑按钮"
- 兼容原生自带工具栏（OPPO v2 等），自动避让
- 工具栏：B / I / U / 高亮 / 字号 7 档 / 字色 8 预设 + 自定义 / Undo / Redo / 批注
- 图片：点击选中 → 替换 / 粘贴 / 删除 / 拖入 / ⌘V
- 拖拽移动文件（v1.7.0）
- 收藏 + 排序 + 搜索（v1.6.0）

❌ **还差**：
- **文本布局调整**：行距、段距、缩进、对齐（现有 Undo/Redo 实际一碰就散架）
- **图片缩放/拖动**：还只能换图不能调大小
- **表格编辑**：加行/删行/合并单元格完全没有
- **列表/引用/标题级别切换**：靠浏览器原生 `document.execCommand` 默认行为，不稳

### 2.2 G2「零数据丢失」完成度：**90%**

✅ **已做到**：
- 停手 2s 自动快照到 `_auto-save/`
- 覆盖前自动备份 `pre-overwrite`
- 三选项保存（覆盖/另存/丢弃）
- 快照 20 份滚动保留
- 跨目录拖拽时同步移动快照

❌ **还差**：
- **没有"历史版本时间线"UI**，快照文件夹要用 Finder 打开看
- **没有 diff 视图**，对比两个版本看不到改动点
- **没有"回到某个历史版本"的一键恢复**，要手动复制粘贴

### 2.3 G3「零 AI 依赖」完成度：**50%**

✅ **已做到**：
- 文字小改（错字、字号、颜色）完全不用回 AI
- 图片替换（换一张图）完全不用回 AI

❌ **还差**：
- **改样式**：想把"某个卡片的背景色"从蓝改成金，不会编辑器里选到 CSS → 还是要回 AI
- **改布局**：两个段落顺序调换、删除某个 section → 只能靠 contenteditable 的删除（容易误伤）
- **改结构**：加一个新章节、加一张表格 → 完全靠 AI

### 2.4 **核心痛点 · 用户未明说但客观存在**

从过去 10 个版本的迭代线看，有 3 个**反复踩的坑**说明产品在这些地方"看起来行，实际不稳"：

| 痛点 | 症状 | 根因 |
|---|---|---|
| **P1 · contenteditable 不稳定** | v1.8.0/1/2 连续三版修字号字色选区问题 | 浏览器原生 contenteditable 在 AI 生成的复杂 HTML 上"水土不服"（嵌套 div、inline style 过多） |
| **P2 · 缓存/版本漂移** | v1.7.1/4 连续改缓存，还出过 CHANGELOG 双写漂移 | 静态资源缓存策略不统一；架构里存在"同一信息两份"的设计债 |
| **P3 · 侧边栏布局累计复杂** | v1.6.1/1.7.1/1.7.4 连续改折叠策略 | 信息密度 vs 展开深度的矛盾随文件数量增长暴露 |

## 3. 距离目标还有多远 · 量化判断

### 3.1 能力覆盖率视角

把"在 DocCenter 里想做的编辑动作"拆成 30 个原子操作，打覆盖率：

```
文字类（10 项）：改错字/加粗/斜体/下划线/字号/字色/高亮/批注/删段/插段
  ✅ 覆盖 9/10，仅"插段"（新起一段）不稳 → 90%

图片类（6 项）：替换/删除/粘贴/拖入/缩放/拖动位置
  ✅ 覆盖 4/6 → 67%

布局类（6 项）：行距/段距/缩进/对齐/段落顺序调换/删 section
  ❌ 覆盖 0/6 → 0%

样式类（4 项）：改背景色/改圆角/改阴影/改字体
  ❌ 覆盖 0/4 → 0%

结构类（4 项）：加表格/加列表/加章节/加分栏
  ❌ 覆盖 0/4 → 0%
```

**总覆盖率：13/30 ≈ 43%。**

### 3.2 使用频率加权

不同能力的使用频率差很大。按"实际会不会改这个"加权：

| 能力 | 原子覆盖 | 使用频率权重 | 加权分 |
|---|---|---|---|
| 文字改错字/加粗等 | 90% | 35% | 31.5 |
| 图片替换/删除 | 67% | 15% | 10.0 |
| 批注/高亮审阅 | 100% | 10% | 10.0 |
| 布局调整 | 0% | 15% | 0 |
| 样式微调 | 0% | 10% | 0 |
| 结构变更 | 0% | 15% | 0 |

**加权能力分：51.5 / 100。**

### 3.3 结论

> **DocCenter v1.8.2 ≈ "及格偏上，但卡在第二关"**
>
> - **第一关（看）**：100% 完成。目录树 + iframe + 自动快照已经很好
> - **第二关（微改）**：~70% 完成。文字+图片+批注能覆盖大多数场景
> - **第三关（调版）**：0%。布局/样式/结构都得重新想路子
>
> **达到北极星 G3 还需至少 2 个大版本（v1.9 / v1.10）**

## 4. 下一步优化路线 · 四条候选路径

### 路径 A · 「编辑稳定性」加固（防御型）

**目标**：解决 P1-P3 三个累计技术债，不加新功能

具体：
1. 引入 Lexical 或 ProseMirror 内核替换 contenteditable（解决 P1 根因）
2. 统一"信息单源原则"审计一遍代码（消灭 CHANGELOG 式漂移）
3. 侧边栏改"虚拟滚动 + 分组折叠"（为未来 1000+ 文件做准备）

**优劣**：
- ✅ 解决多年技术债，后面不会再踩
- ❌ 用户看不到肉眼可见的新功能，2-3 周纯投入
- ❌ 换编辑内核风险高，可能引发新 bug

**适合时机**：准备冲刺下一波大版本前的"固本阶段"

---

### 路径 B · 「版本时间线 + diff」（G2 闭环）

**目标**：把 `_auto-save/` 从"文件夹"升级成"产品"

具体：
1. 侧边栏点文件 → 右侧面板 Tab "历史版本"
2. 时间线卡片：每张 = 一个快照 + 时间 + 一行变更 summary（用 diff 算字符数）
3. 点卡片 → 右侧 split-view，左当前 右历史，滚动同步
4. "一键恢复"按钮，恢复前自动再快照一次当前

**优劣**：
- ✅ 解决"快照用不上"的尴尬（目前功能存在但没人看）
- ✅ 用户肉眼可见的大功能，正反馈强
- ✅ 技术难度中等（diff-match-patch 库成熟）
- ❌ 不解决编辑不稳的根本问题

**适合时机**：**本轮最推荐** —— 收益高、风险低、闭环 G2

---

### 路径 C · 「样式检查器」（G3 进阶）

**目标**：让用户能改"卡片背景色""字体"这类样式

具体：
1. 点任意元素 → 右侧弹 DevTools 风的样式检查器
2. 常用属性变拖杆/色板：`background-color / color / font-size / padding / border-radius / box-shadow`
3. 改哪个元素自动加 `style="..."`，不污染原 CSS
4. "查看生成的 CSS"可展开

**优劣**：
- ✅ 解决 G3 里"改样式"的 40% 场景
- ✅ 用户价值极高（AI 生成的卡片颜色不满意是最常见抱怨）
- ❌ 技术难度高（要做 DOM 选择器 + live preview + serialize 回写）
- ❌ UX 很难做得好（DevTools 复杂度 vs Notion 简洁度的 tradeoff）

**适合时机**：路径 B 之后，作为 v1.10 大版本

---

### 路径 D · 「AI-assisted Edit」（护城河型）

**目标**：用 AI 辅助编辑，但**本地化**

具体：
1. 选中任意元素 → 右键"✨ AI 改一下"
2. 弹对话框：「想改成什么样？」→ "红色背景改成金色渐变" / "字再小一点"
3. 调本地模型（Ollama qwen2.5）或远程（仅用户自配 key）→ 返回新的 inline style
4. Diff 预览 → 接受/拒绝

**优劣**：
- ✅ 差异化极强（没有竞品做"本地+轻量"的 AI 编辑）
- ✅ 和"AI 生成报告"的上游自然衔接
- ❌ 工程复杂度最高
- ❌ 用户需要配 AI key 或装 Ollama，门槛上升
- ❌ 改的不满意会让用户怪产品而不是 AI

**适合时机**：v2.0 战略级，需要先把 G1/G2 打磨到位

---

## 5. 推荐优先级

```
本周（v1.9.0）：路径 B · 版本时间线 + diff  ← 闭环最大痛点 G2
下周（v1.9.5）：路径 A 中的 #2「信息单源审计」 + 侧边栏优化
再下个月（v1.10）：路径 C · 样式检查器 MVP
v2.0（6 周后）：路径 D · 本地 AI 辅助编辑
```

### 为什么推 B 不推 A

1. **路径 A 纯投入**，用户看不到；路径 B 有可视化产出
2. **路径 B 是 DocCenter 独有价值点**——AI 生成报告 + 自动快照的组合天然适合时间线
3. **路径 B 难度中等**，1-2 天能出 MVP，不耽误后面推其他路径
4. **路径 A 的最大痛点（contenteditable 不稳）在 B 做完后再定决策更合理**——届时用户反馈更集中，知道该换哪个内核

## 6. v1.9.0 · 版本时间线功能规格草案

### UX

```
┌─── DocCenter ─────────────────────────────────────────────────┐
│ [目录树]    │ [iframe 当前文件]             │ [历史 Tab]       │
│            │                                │ ──────────      │
│  文件 A    │                                │ ⏱ 10:23 · 现在  │
│  文件 B    │   〔HTML 正文渲染〕              │ ⏱ 10:21 · 2min │
│  文件 C ●  │                                │   ➕12 ➖3 字    │
│   └─ ....  │                                │ ⏱ 10:15 · 8min │
│            │                                │   ➕45 字图*1    │
│            │                                │  [▶ 预览] [↩ 恢复]│
└────────────┴────────────────────────────────┴──────────────────┘
```

### 技术方案

- 时间线数据源：现有 `_auto-save/` 文件夹，按 mtime 排序
- Diff 算法：`diff-match-patch.js`（Google 开源，8KB，成熟）
- 变更 summary：`+N 字 / -M 字 / 图*K / 样式变更`
- 预览用 `<iframe srcdoc="..." sandbox>` 渲染历史版本
- 恢复逻辑：先把当前状态写一份"pre-restore"快照，再把历史版覆盖回源文件

### Server 新增 API

```
GET  /api/history?path=<relpath>        # 返回快照列表
GET  /api/history/diff?path=&v1=&v2=    # 返回两个版本的 diff 结构
POST /api/history/restore?path=&to=     # 恢复到指定快照
```

### 验收清单

- [ ] 时间线正确显示所有快照
- [ ] diff 摘要数字准确
- [ ] 预览不污染当前编辑状态
- [ ] 恢复前强制再快照一次
- [ ] 跨文件切换时时间线自动刷新

## 7. 一句话总结

> **v1.8.2 是一个"看得见编辑能力"的阶段性终点。v1.9 应该转向"看得见历史"——把快照从底层能力升级成用户可感知的产品。然后再考虑样式/AI 这些更大动作。**

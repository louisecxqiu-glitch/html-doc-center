BUSINESS MODEL

双轨制：本地开源 + 云端 SaaS

本地版不限制在线分享——让用户先体验价值，再引导付费升级

本地版 · 开源

流量入口 + 社区资产

  • 编辑器全功能（contentEditable）
  • 离线分享（自包含 HTML 导出）
  • 可自建分享服务（Docker）
  • 永远免费，开源透明
  • 不限制在线分享功能——让用户体验到价值
云端版 · SaaS

增值服务 + 商业化主体

  • 托管分享（免运维）
  • 品牌定制（Pro 去水印）
  • 协作编辑（Team）
  • 企业私有部署（Enterprise）
  • 按层收费——Free / Pro / Team / Enterprise

核心设计：本地版和云端版功能不重叠——本地版做"创作"，云端版做"分发与协作"。用户不会问"我为什么不自己跑本地版"，因为本地版不提供"分发与协作"能力。愿意折腾 = 免费（自建 Docker）；不愿意折腾 = 付费用云端版。

TECHNICAL ARCHITECTURE

4 阶段技术演进路线图

每阶段 2-4 周，渐进式叠加，不推翻重来

Phase 1 · 已完成

本地 HTML Studio (Python) ├─ 编辑器 (saver-runtime.js) ├─ 离线分享 (export-share API) └─ 在线分享 → CloudBase 单环境 ├─ 云函数 └─ NoSQL: shares

Phase 2 · Pro（1-2 周）

本地版 + config.json token │ ▼ CloudBase /test/* ├─ AuthMiddleware（识别 uid/匿名） ├─ QuotaMiddleware（校验月度额度） ├─ WatermarkMiddleware（免费注入水印） └─ NoSQL: shares + users + usage_logs

Phase 3 · Team（1 月）

新增： ├─ teams 集合 + team_members ├─ /api/team/* 路由 ├─ WebSocket 实时协作 └─ 团队素材库（共享图片/模板）

Phase 4 · Enterprise（2-3 月）

企业独立 CloudBase 环境 ├─ audit_logs（审计日志） ├─ sso_configs（SSO/LDAP/SAML） ├─ 企业插件包 ├─ Docker Compose 一键部署 └─ 本地版指向企业环境

架构原则：① 渐进式——不改本地版代码，云端版加层 ② 匿名优先——不登录也能用，登录后解锁 Pro ③ 中间件模式——Auth/Quota/Watermark 不侵入业务逻辑 ④ 配置驱动——本地版 config.json 的 share_server 指向哪个后端就哪个

RISK ASSESSMENT

6 项风险 · 高 2 / 中 3 / 低 1

每项都有明确的缓解方案

CloudBase 单点依赖高
Pro/Team 全押 CloudBase，腾讯调价/停服/故障全盘受影响。
缓解：云函数代码标准 Node.js 可迁移；本地版 share_server.py 是独立 Python 实现，可随时切换后端。
创始人单点投入高
当前只有创始人一人开发 + 运营，精力瓶颈明显。
缓解：Phase 2 后评估招第一个工程师；开源社区贡献者补充；非核心工作外包。
开源版被 fork 做竞品 SaaS中
本地版开源全功能，竞品可 fork 后自建 SaaS 抢市场。
缓解：云端版核心（用户系统/计费/协作/统计）不开源，竞品 fork 拿到的只是编辑器。
付费转化率低于 3%中
免费版体验太好，用户没动力升级 Pro。
缓解：水印是核心 monetization——内容创作者对"分享页带别人 logo"高度敏感。若转化低可调水印强度。
企业版合规要求中
政企客户可能要求数据不出境、等保三级、国密算法。
缓解：Enterprise 只做私有部署（数据在企业内网），不碰 SaaS 合规。
微信支付审核周期低
微信支付商户号申请可能 1-2 周。
缓解：Week 1 就提交申请，Week 2 开发并行等待；或先用 Stripe 兜底。
